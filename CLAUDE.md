# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Development
bun run dev          # Generates Payload import map, then starts Next.js dev server (http://localhost:3000)
bun run build        # Production build
bun run start        # Start production server

# Code quality
bun run lint         # ESLint

# Testing (vitest — independent of Next.js)
bun run test         # Run tests once
bun run test:watch   # Watch mode
bunx vitest run src/test/example.test.js  # Single test file

# Database
bun run seed         # Seed DB with default users (runs src/payload/run-seed.ts)
```

The project uses **bun** as the package manager. Prefer `bun` over `npm`/`npx`.

## Environment

Required `.env.local` variables:
- `NEXT_PUBLIC_APP_URL` — e.g. `http://localhost:3000`
- `DATABASE_URI` — PostgreSQL connection string (e.g. `postgresql://postgres:root@localhost:5432/hackifytech`)
- `PAYLOAD_SECRET` — Secret key for Payload CMS (change in production)

## Architecture

### Tech Stack
- **Next.js 15** App Router + **React 18**
- **Payload CMS v3** — headless CMS + admin panel at `/admin`, PostgreSQL-backed
- **Tailwind CSS** + **shadcn-ui** (`src/components/ui/`) for all UI
- **next/font** (Space Grotesk + Inter) — loaded in root layout, no Google Fonts @import
- **TanStack Query v5** for data fetching
- **framer-motion** for animations
- **react-hook-form + zod** for forms
- **@react-pdf/renderer** for PDF generation (syllabus downloads)

Route groups (`(public)`, `(auth)`, `(payload)`) do not affect URLs — they only control layout nesting.

### SEO Pattern

Every `app/**/page.jsx` is a **server component** that:
1. Exports `metadata` (or `generateMetadata` for dynamic routes) — this is what search engines see
2. Renders the corresponding `src/views/` client component

```jsx
// app/(public)/about/page.jsx  (server component — no "use client")
import AboutPage from "@/views/AboutPage";

export const metadata = {
  title: "About HackifyTech",
  description: "...",
  openGraph: { ... },
};

export default function Page() {
  return <AboutPage />;
}
```

Dynamic routes use `generateMetadata({ params })` instead of a static export.

### src/views/ Components

All page content lives in `src/views/`. These are **client components** (`"use client"`) because they use framer-motion, useState, useParams, etc. The `app/` route files are thin server shells that only contribute SEO metadata.

The directory is named `views/` (not `pages/`) to avoid conflicting with Next.js's Pages Router detection of a `pages/` directory.

**Do not** add business logic or data fetching to `app/**/page.jsx` files — keep them as thin wrappers.

### Payload CMS

The backend is **Payload CMS v3** with a PostgreSQL database. Config entry point: `payload.config.ts` at the project root.

**Admin UI:** `http://localhost:3000/admin`

**Collections** (defined in `src/payload/collections/`):
- `Users` — Auth + role management (super-admin, admin, trainer, student, placement)
- `Courses` — Course catalog with pricing, syllabus, rich content
- `Batches` — Training batch scheduling and enrollment
- `Attendance` — Student attendance records
- `Assignments` — Course assignments
- `Leads` — Inquiry/lead management
- `Testimonials` — Student testimonials
- `Media` — File uploads (20 MB limit)

**Auto-seeding:** On first launch with an empty database, `payload.config.ts` automatically calls `seedUsers()` to create default accounts. Run `bun run seed` to trigger manually.

**Auto-generated files — do not hand-edit:**
- `payload-types.ts` (root) — TypeScript types generated from collection schemas; regenerated on build/dev
- `app/(payload)/admin/importMap.js` — Generated by `payload generate:importmap` (runs automatically in `bun run dev`)

### Auth & Context

`src/contexts/AuthContext.jsx` is the single source of truth for auth state. It exposes:
- `user`, `profile`, `role`, `loading`, `profileLoading`
- `profileComplete`
- `signIn()`, `signOut()`, `refreshProfile()`

Role priority: `super-admin > admin > trainer > student > placement`. Managed via Payload's `Users` collection.

### Path Aliases

`@/` maps to `src/`. Used throughout the codebase.

`@payload-config` maps to `./payload.config.ts` — used internally by Payload.

### shadcn-ui Components

All primitives live in `src/components/ui/`. Add new components via:
```bash
bunx shadcn-ui add <component>
```

Composed app-level components (`Navbar`, `Footer`, `LoginDialog`, etc.) live in `src/components/`.

### Static vs Dynamic Data

Public-facing pages (CoursesPage, HomePage, etc.) use hardcoded data arrays in `src/data/`. Portal/dashboard pages are expected to query the Payload REST API via TanStack Query. `src/App.jsx` is dead code — Next.js uses `app/layout.jsx` as the entry point.
